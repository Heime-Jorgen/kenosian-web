Overview

In prefill-decode (PD) disaggregated LLM inference, the prefill GPU processes the prompt and generates a KV cache — a large tensor structure that must be transferred to one or more decode GPUs before generation can begin. At scale, this KV transfer is the dominant bottleneck: a 70B model serving a 1K-token prompt generates ~313MB of KV data per request. On commodity networking, this stalls decode and collapses throughput under load.

Hydra-KV eliminates this bottleneck by intercepting the outbound KV transfer at the prefill node, applying proprietary lattice compression (3.20× reduction), then checking a CRDT deduplication index to skip any blocks the decode node already holds from previous sessions (95.6% of blocks in typical workloads). Only novel, compressed deltas travel over the wire — via QUIC with TLS 1.3.

The 3-step process: prefill sidecar intercepts and compresses → GapVec5 routing selects the optimal decode node based on 5 real-time network dimensions → decode sidecar decompresses in-place and hands standard KV tensors to the decode model. No model changes. No infrastructure overhaul. The decode GPU sees an ordinary KV cache — only the transfer layer changes.

Architecture

  • · Prefill node compresses KV cache with proprietary lattice algorithm
  • · Sidecar runs alongside vLLM/SGLang with no model changes (50-line drop-in connector)
  • · GapVec5 uses 5 dimensions: ISP density, latency, packet loss, throughput, DPI fingerprint
  • · QUIC transport with TLS 1.3 — no plaintext KV in transit
  • · Decode node verifies integrity via TTTPS seal on each chunk
  • · GapVec5 selects optimal transfer path (ISP latency, loss rate, throughput)
  • · CRDT deduplication eliminates redundant KV transfer across sessions — if the same prompt prefix was seen yesterday, the KV cache is reused today

How It Works — Step by Step

  1. 1. Prefill node processes prompt — generates KV cache tensors.
  2. 2. Hydra-KV sidecar intercepts outbound KV transfer, compresses with proprietary lattice algorithm (3.20× reduction).
  3. 3. CRDT deduplication checks if identical blocks were seen before — 95.6% of blocks are already on the decode node.
  4. 4. GapVec5 probes all decode nodes in real-time, selects optimal path based on 5 network dimensions: ISP density, latency, packet loss, throughput, DPI fingerprint.
  5. 5. QUIC transport delivers compressed, deduplicated chunks to decode sidecar.
  6. 6. Decode sidecar decompresses in-place — decode model sees standard KV cache, no changes needed.
  7. 7. Each chunk sealed with TTTPS token (if Hydra-Audit is enabled) — full cryptographic audit trail.

Hydra-KV + Audit Integration

When Hydra-KV is paired with Hydra-Audit, every KV cache chunk transfer is sealed with a TTTPS temporal token. This creates a cryptographic audit trail of all inter-node data movement: which prefill node sent what to which decode node, at what time, in what order.

Attack TypeDescriptionAuto-Logged
REPLAY Same KV chunk retransmitted — indicates replay injection attempt Yes
ORDERING Out-of-sequence chunk delivery — indicates tampering with transfer order Yes
FORGE Tampered chunk content — TTTPS seal mismatch detected at decode node Yes

Audit log is queryable per context pool: GET /audit?ctx_id=<pool-id> — returns per-chunk delivery proof, Byzantine event count, and timing data. Example: a latency spike at a specific decode node can be cross-referenced against the TTTPS audit log to determine whether it was caused by a replay attack or an ordering violation — not a network issue. This is especially valuable in multi-tenant inference clusters where node compromise is a real threat.
