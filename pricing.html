OpenTTT — Solving Workflow Amnesia for Claude Code

Every Claude Code long-horizon workflow hits the same wall: context compression erases action history. Agent B doesn't know what Agent A decided. Agents duplicate work, corrupt state, lose decisions made hours earlier.

OpenTTT anchors every workflow step to a cryptographic timestamp on an external server — physically separate from Claude's context window. When compression hits, agents call pot_query(eventId) for O(1) exact step recall. Zero lost state.

Mathematical guarantees: SHA-3 256-bit eventId: collision probability 2⁻²⁵⁶ · O(1) exact lookup · DAG causal chain via prevEventId · Ed25519 non-repudiation · Golay shards: ≥97% at BER=0.05 · Redis AOF persistence, 90-day TTL

7 tools: pot_generate · pot_verify · pot_query · pot_graph · pot_checkpoint · pot_stats · pot_health  ·  IETF TTTPS-standardized · MiFIR Art.22c audit-grade

Example: A 20-agent Dynamic Workflow refactors a 500K-line codebase over hours. After each context compression, every agent calls pot_query(eventId) — the external server returns its exact action history. Zero lost state. Zero duplicate work.

Why OpenTTT — Not Letta or Mem0

Tool Memory model Hot-path cost Recall type
Letta / Mem0 Semantic RAG embedding + vector search per op Fuzzy similarity
LangGraph / LangMem Graph state + semantic checkpoint I/O + embeddings Semantic + graph
OpenTTT / ttt-mcp Cryptographic event log sign + hash + write — 0 embedding calls O(1) exact hash lookup

OpenTTT stamps only the steps worth checkpointing — not every query, not every token. At scale, eliminating the embedding layer is a structural cost advantage, not a feature trade-off. For fuzzy semantic search, pair with a vector DB. For deterministic state recovery that survives context compaction: OpenTTT.
